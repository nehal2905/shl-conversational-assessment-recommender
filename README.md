# SHL Conversational Assessment Recommender

A **stateless conversational AI API** that guides recruiters from vague hiring intents (e.g., *"I'm hiring a Java developer"*) to grounded recommendations of **SHL Individual Test Solutions**.

The agent follows a conversational workflow to:

- Clarify vague hiring requirements
- Recommend **1–10** relevant SHL assessments
- Refine recommendations when requirements change
- Compare SHL assessments using catalog evidence
- Refuse off-topic and prompt-injection requests

Every assessment name and URL returned by the system is **grounded directly from the SHL catalog**. The language model never generates assessment names or URLs on its own.

# System Architecture

```
POST /chat

START
   │
   ▼
Analyze
   │
   ├── Clarify
   ├── Recommend
   ├── Compare
   └── Refuse
        │
        ▼
     Format
        │
        ▼
       END
```

## Components

### Analyze

- Stateless analysis over the complete conversation history
- Structured reasoning using Groq
- Slot extraction (role, seniority, skills, constraints)
- Automatic routing decision

### Clarify

- Requests only the missing hiring information
- Stops clarifying once sufficient context has been gathered
- Limited clarification turns to prevent endless conversations

### Recommend

Hybrid retrieval pipeline:

- BM25 sparse retrieval
- FAISS dense retrieval
- Normalized score fusion
- Cosine reranking
- Grounding against catalog IDs

### Compare

- Resolves assessment names using RapidFuzz
- Produces grounded comparisons directly from catalog metadata

### Refuse

Safely rejects:

- Prompt injection
- General career advice
- Legal questions
- Off-topic requests

### Format

Produces the required API schema while ensuring every response remains valid.

# Engineering Invariants

The implementation enforces several architectural guarantees.

- Assessment names are never generated by the LLM.
- Assessment URLs always come directly from the SHL catalog.
- The API is fully stateless.
- The complete conversation history is reprocessed on every request.
- Hybrid retrieval is always used.
- Responses always follow the required API schema.
- Prompt injection attempts are treated as data rather than instructions.
- Clarification is bounded to avoid unnecessary conversation loops.

## GET /health

Returns:

```json
{
  "status": "ok"
}
```

Used for readiness checks.

---

## POST /chat

The API is **stateless**.

Every request must include the complete conversation history.

### Example Request

```json
{
  "messages": [
    {
      "role": "user",
      "content": "I'm hiring a mid-level Java backend developer with Spring Boot and SQL."
    }
  ]
}
```

---

## Response Schema

```json
{
  "reply": "string",
  "recommendations": [
    {
      "name": "string",
      "url": "string",
      "test_type": "K"
    }
  ],
  "end_of_conversation": false
}
```

- `recommendations` is empty while clarifying or refusing.
- Once enough context exists, recommendations contains **1–10** assessments.
- `end_of_conversation` indicates whether the recommendation task is complete.

# Quick Start

## Create a virtual environment

```bash
python -m venv .venv
```

Windows

```powershell
.\.venv\Scripts\Activate.ps1
```

## Install dependencies

```bash
pip install -r requirements.txt
```

## Configure environment

```bash
cp .env.example .env
```

Add your API key:

```
GROQ_API_KEY=your_key
```

## Build retrieval indexes

```bash
python scripts/build_index.py
```

## Run locally

```bash
uvicorn app.main:app --reload
```

# Offline Mode

If `GROQ_API_KEY` is not configured, the application automatically switches to a deterministic offline implementation.

This enables:

- Local development
- Automated testing
- Continuous Integration
- Offline evaluation

without requiring an external LLM.

# Data

The system operates over the **SHL Individual Test Solutions** catalog.

The committed repository currently contains **526 SHL assessment entries**.

To rebuild the catalog from the latest publicly available SHL catalog:

```bash
playwright install chromium

python scripts/scrape_catalog.py

python scripts/build_index.py
```

Generated retrieval indexes (`faiss.index`, `bm25.pkl`, and `ids.json`) are automatically rebuilt during Docker image creation and are intentionally excluded from version control.

# Evaluation

Evaluation measures four aspects of the system:

- Retrieval quality (Recall@10)
- Recommendation relevance
- Groundedness (Hallucination Rate)
- Conversational behavior

Run the complete evaluation suite:

```bash
python -m pytest -q
```

```bash
python eval/probes.py
```

```bash
python eval/replay.py
```

The public conversation traces can be placed inside `eval/traces/` to reproduce replay evaluation.
